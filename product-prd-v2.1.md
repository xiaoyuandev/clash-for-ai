# Clash for AI PRD v2.1

## 1. 文档定位

本版本不是概念方案，而是面向开发落地的执行版 PRD。

目标是明确三件事：

1. 这个产品到底解决什么问题。
2. 首版到底承诺什么，不承诺什么。
3. 工程实现时哪些地方必须保守，不能靠乐观假设。

---

## 2. 产品概述

### 2.1 一句话定义

Clash for AI 是一个面向 AI 重度用户的本地桌面工具，用于管理多个中转 API Provider，提供统一本地接入地址、Provider 切换、健康检查、基础观测和故障排查能力。

### 2.2 产品本质

一个本地 Provider 管理面板 + 一个本地 API Gateway。

用户在各类 AI 工具中只需要配置一次本地地址：

`http://localhost:3456/v1`

后续切换不同 Provider、查看健康状态、定位故障原因，都在 Clash for AI 中完成，而不是反复修改每个工具的 `base_url` 和 `api_key`。

### 2.3 产品目标

1. 用户在多个 AI 工具中只配置一次本地入口。
2. 用户可以在 1 个界面里管理多个 Provider。
3. 切换 Provider 不需要重新改工具配置。
4. 请求失败时，用户可以快速定位问题大类。
5. 产品首版以“稳定透传 + 可观测”优先，而不是“做最强协议兼容层”。

### 2.4 产品非目标

首版不追求以下目标：

1. 不做官方模型厂商统一适配层。
2. 不做系统级代理或 VPN。
3. 不做云端中转服务。
4. 不做企业协作、权限、审计。
5. 不承诺兼容所有 Provider 与所有客户端组合。
6. 不做智能路由、自动 fallback、负载均衡。

## 3. 用户与场景

### 3.1 核心用户

同时使用两家及以上中转 Provider 的开发者、独立开发者和 AI 重度用户。

### 3.2 典型场景

1. 主力 Provider 不稳定，需要快速切换到备选 Provider。
2. 某个 Provider 额度不足，需要临时切换。
3. 新接入一个 Provider，想先验证连通性、测速和基础兼容性。
4. 同时使用 Claude Code、Cursor、Chatbox、Cherry Studio 等工具，不希望反复改配置。

### 3.3 不服务的用户

1. 只使用单一官方 API，且没有切换需求的用户。
2. 需要团队级权限管理和审计的组织用户。
3. 期望通过本产品解决所有模型协议差异的用户。

---

## 4. 核心问题

当前目标用户主要有四个痛点：

| 问题 | 当前表现 | 结果 |
|------|----------|------|
| Provider 切换成本高 | 每次切换都要改多个工具配置 | 容易出错，切换慢 |
| 状态不可见 | 不知道哪家可用、哪家慢、哪家余额不足 | 决策依赖经验和试错 |
| 故障归因困难 | 失败后分不清是本地问题、鉴权问题还是上游问题 | 排障成本高 |
| 配置分散 | 多个 key 和 base_url 散落在不同工具里 | 管理混乱，泄漏风险高 |

## 5. 产品定位与边界

### 5.1 产品定位

Clash for AI 的定位不是“另一个中转平台”，而是“中转平台消费者的本地控制台”。

它的核心价值有两个：

1. 统一接入：所有工具固定连接本地 Gateway。
2. 统一观测：Provider 状态、错误和切换都在本地控制台完成。

### 5.2 明确边界

首版只做以下能力：

1. Provider 管理。
2. 本地 Gateway。
3. 手动切换当前活跃 Provider。
4. 健康检查和基础测速。
5. 请求日志。
6. 保守的错误分类。
7. 支持少量已适配 Provider 的余额查询。

首版明确不做：

1. 自动协议转换。
2. 自动接管用户工具配置。
3. 自动 fallback。
4. 模型级智能路由。
5. 订阅协议生态。

---

## 6. 关键产品假设

### 6.1 可以成立的假设

1. 目标用户愿意把工具的 `base_url` 改成一次 `localhost`。
2. 目标用户愿意把敏感 key 集中保存在本地桌面工具里。
3. 大部分目标用户的核心需求是“切换 + 状态透明”，而不是“复杂路由策略”。

### 6.2 不能写成默认事实的假设

以下内容不能当作产品默认事实，只能当作“按 Provider 逐步验证”的现实条件：

1. 不是所有 Provider 都同时稳定支持 OpenAI-compatible 和 Anthropic-compatible 接口。
2. 不是所有 Provider 都接受相同的 Auth Header。
3. 不是所有 Provider 都支持 `/v1/models`。
4. 不是所有 Provider 都支持 SSE，或者 SSE 行为完全一致。
5. 不是所有 429 都代表额度不足。

因此首版必须采用两个原则：

1. 产品承诺“尽量透传并提供能力声明”，不承诺“天然兼容一切”。
2. 所有兼容性都应通过验证矩阵管理，而不是写成口头前提。

---

## 7. 用户价值主张

相较于手动管理 Provider，本产品的核心提升不是“少点几下”，而是：

1. 一个固定入口，所有工具复用。
2. Provider 切换集中完成。
3. 请求问题可以从“完全黑盒”变成“至少知道问题大类”。
4. 用户对各 Provider 的健康状态有持续感知。

## 8. 核心能力设计

### 8.1 Provider 管理

### 8.1.1 功能

1. 添加 Provider。
2. 编辑 Provider。
3. 删除 Provider。
4. 设置活跃 Provider。
5. 查看 Provider 健康状态。
6. 查看 Provider 能力声明。
7. 拉取模型列表。
8. 对已适配 Provider 查询余额。

### 8.1.2 Provider 数据模型

```json
{
  "id": "uuid",
  "name": "Provider A",
  "base_url": "https://api.provider-a.com/v1",
  "api_key_ref": "credential://provider/a",
  "auth_mode": "bearer",
  "extra_headers": {
    "anthropic-version": "2023-06-01"
  },
  "capabilities": {
    "supports_openai_compatible": true,
    "supports_anthropic_compatible": true,
    "supports_models_api": true,
    "supports_balance_api": false,
    "supports_stream": true
  },
  "status": {
    "is_active": false,
    "last_healthcheck_at": "2026-04-17T12:00:00Z",
    "last_health_status": "ok"
  },
  "created_at": "2026-04-17T12:00:00Z",
  "updated_at": "2026-04-17T12:00:00Z"
}
```

### 8.1.3 字段说明

1. `base_url` 由用户填写，通常建议以 `/v1` 结尾，但不强制。
2. `api_key_ref` 不直接保存明文 key，而是保存本地安全存储引用。
3. `auth_mode` 支持：
   - `bearer`
   - `x-api-key`
   - `both`
4. `extra_headers` 用于兼容少量必须额外注入的固定 Header。
5. `capabilities` 是产品内部的能力声明，不代表对所有客户端都兼容，只代表该 Provider 经验证具备这些能力。

### 8.1.4 交互要求

1. 添加 Provider 后自动触发健康检查。
2. 若 `base_url` 不以 `/v1` 结尾，显示提醒但允许保存。
3. 删除当前活跃 Provider 时，要求先切换到其他 Provider。
4. 活跃 Provider 切换后立即生效，无需重启应用。

## 8.2 本地 Gateway

### 8.2.1 功能目标

本地 Gateway 的职责不是协议转换，而是稳定做四件事：

1. 接收本地客户端请求。
2. 根据当前活跃 Provider 转发请求。
3. 注入正确的鉴权和固定 Header。
4. 记录请求元数据和结果。

### 8.2.2 输入输出

本地暴露统一入口：

`http://localhost:3456/v1`

首版默认只处理 `/v1/*` 路径。

### 8.2.3 路径转发规则

规则如下：

1. 本地入口固定以 `/v1` 开头。
2. 转发时移除请求路径中的 `/v1` 前缀。
3. 将剩余路径拼接到 Provider `base_url` 后。

示例：

| Provider base_url | 本地请求 | 上游请求 |
|-------------------|----------|----------|
| `https://a.com/v1` | `/v1/messages` | `https://a.com/v1/messages` |
| `https://b.com` | `/v1/messages` | `https://b.com/messages` |

### 8.2.4 Header 注入策略

这部分必须做成可配置，不允许写死。

按 `auth_mode` 处理：

1. `bearer`
   - 注入 `Authorization: Bearer <api_key>`
2. `x-api-key`
   - 注入 `x-api-key: <api_key>`
3. `both`
   - 同时注入以上两个 Header

同时注入该 Provider 配置的 `extra_headers`。

外部工具传入的认证 Header 默认会被覆盖，以避免用户本地工具配置与 Provider 实际 key 不一致。

### 8.2.5 协议策略

首版采用“透传优先，最小解析”的策略：

1. 不做协议转换。
2. 不重写请求 body。
3. 不依赖 body 解析才能完成转发。
4. 对 JSON 请求可做浅解析，最佳努力提取 `model` 字段用于日志。
5. 解析失败不能影响请求正常转发。

### 8.2.6 Stream 支持

1. 首版必须支持透传 SSE。
2. 对 SSE 只做最小干预，不做消息重组。
3. 日志只记录是否 stream、首字节耗时和总耗时，不记录完整流内容。

### 8.2.7 异常处理

若当前没有活跃 Provider，Gateway 直接返回本地错误，明确告知用户未配置可用 Provider。

## 8.3 健康检查与测速

### 8.3.1 产品原则

“健康检查”和“测速”不是一回事，首版必须拆开。

### 8.3.2 健康检查

目标是低成本判断该 Provider 当前是否基本可用。

检查策略优先级：

1. 优先请求轻量级接口，如 `/models`。
2. 若 Provider 不支持 `/models`，可使用配置化探测路径。
3. 仅在必要时使用小成本文本生成请求。

输出结果：

1. 可连接 / 不可连接
2. HTTP 状态码
3. 基础延迟
4. 失败原因摘要

### 8.3.3 测速

测速是可选操作，不应默认高频执行。

测速指标：

1. DNS + 连接耗时
2. 首字节耗时
3. 首 token 耗时
4. 完整响应耗时

若测速会真实产生 token 消耗，UI 必须明确提示“可能产生少量费用”。

### 8.3.4 批量测速

支持用户手动触发所有 Provider 批量测速，但不建议自动高频跑批量测速，以避免：

1. 消耗配额
2. 误判短时网络抖动
3. 触发上游限频

---

## 8.4 模型列表

### 8.4.1 目标

为用户提供“该 Provider 当前可见模型”的参考信息。

### 8.4.2 策略

1. 若 Provider 支持 `/v1/models`，则自动拉取。
2. 拉取失败时，明确标记“不支持或暂时不可用”。
3. 支持用户手动维护模型别名或补充列表。

### 8.4.3 边界

模型列表是参考信息，不作为路由依据。

---

## 8.5 余额查询

### 8.5.1 产品策略

余额查询不是通用能力，只对已适配 Provider 提供。

### 8.5.2 分层支持

| 支持级别 | 说明 | 处理方式 |
|----------|------|----------|
| 标准支持 | 已有明确余额接口且已验证 | 自动查询并展示 |
| 适配支持 | 有接口但结构特殊 | 通过 provider adapter 支持 |
| 不支持 | 无公开接口或不稳定 | 显示“不支持” |

### 8.5.3 产品要求

1. 不允许对未适配 Provider 显示伪余额。
2. 不将余额功能定义为首版核心卖点。

---

## 8.6 请求日志与故障排查

### 8.6.1 日志目标

日志的目标不是做审计，而是帮助个人用户快速排障。

### 8.6.2 日志字段

```json
{
  "id": "uuid",
  "timestamp": "2026-04-17T12:00:00Z",
  "provider_id": "uuid",
  "provider_name": "Provider A",
  "method": "POST",
  "path": "/v1/chat/completions",
  "model": "gpt-4.1",
  "status_code": 200,
  "is_stream": true,
  "upstream_host": "api.provider-a.com",
  "latency_ms": 1230,
  "first_byte_ms": 320,
  "first_token_ms": 780,
  "error_type": null,
  "error_message": null,
  "error_snippet": null
}
```

### 8.6.3 日志原则

1. 不记录完整请求 body。
2. 不记录完整响应 body。
3. 允许记录截断后的错误摘要，如前 200 个字符。
4. 默认保留最近 1000 条。
5. 支持按 Provider、状态码、错误类型筛选。

### 8.6.4 错误分类策略

首版错误分类必须保守，避免“错误地精确”。

#### 第一层：通用分类

| 错误类型 | 判断依据 |
|----------|----------|
| `no_active_provider` | 本地未配置活跃 Provider |
| `network_error` | 连接失败、DNS、TLS 错误 |
| `timeout` | 本地超时 |
| `auth_error` | 401/403 |
| `client_error` | 未命中的其他 4xx |
| `rate_limited` | 429，但无法确认是额度还是频控 |
| `upstream_error` | 5xx |
| `unknown_error` | 无法归类 |

#### 第二层：Provider 适配分类

仅对已适配 Provider，在有足够证据时再细分：

1. `quota_exhausted`
2. `model_not_found`
3. `account_suspended`

产品上必须把“通用分类”和“适配分类”区分显示，避免误导用户。

## 9. 兼容性矩阵

这是首版必须存在的文档和测试资产，不是可选项。

### 9.1 目的

定义“哪些客户端 + 哪些 Provider + 哪些协议路径”是实际验证通过的。

### 9.2 首版建议覆盖

客户端维度：

1. Claude Code
2. Cursor
3. Chatbox 或 Cherry Studio

协议路径维度：

1. `/v1/messages`
2. `/v1/chat/completions`
3. `/v1/models`

能力维度：

1. bearer header
2. x-api-key header
3. both
4. stream
5. 非 stream

### 9.3 产品承诺

只有进入兼容性矩阵并完成验证的组合，才能在产品或文档里宣称“已支持”。

## 10. 安全与隐私

### 10.1 API Key 存储

首版不自行设计“硬件信息派生密钥”方案。

采用系统原生安全存储：

1. macOS 使用 Keychain
2. Windows 使用 DPAPI 或 Credential Manager
3. Linux 优先使用 Secret Service，若不可用则提供降级方案

SQLite 中仅存储安全引用、元数据和缓存，不存明文 key。

### 10.2 本地访问安全

1. Gateway 默认仅监听 `127.0.0.1`
2. 不默认监听局域网地址
3. 若后续支持局域网共享，必须单独开启并明确告警

### 10.3 日志隐私

1. 默认不保存 prompt 内容
2. 默认不保存完整响应内容
3. 导出日志时需提示可能包含敏感元信息

## 11. 技术方案

### 11.1 推荐架构

Go 核心服务 + Electron 桌面外壳。

### 11.2 分层职责

| 层 | 技术 | 职责 |
|----|------|------|
| 桌面壳层 | Electron | 窗口、托盘、开机自启、进程管理 |
| UI 层 | React | Provider 管理、状态面板、日志界面 |
| 核心服务层 | Go | Gateway、健康检查、日志、Provider API |
| 存储层 | SQLite + 系统安全存储 | 元数据持久化与 key 管理 |

### 11.3 核心原则

1. 所有网络与业务逻辑在 Go 层。
2. Electron 只负责外壳与 UI，不承担代理逻辑。
3. Gateway 是长期常驻进程，UI 可以关闭，代理仍可运行。

### 11.4 关键实现要求

1. Gateway 需要支持并发请求。
2. Gateway 需要正确透传 stream。
3. 活跃 Provider 切换必须线程安全。
4. 日志写入不能阻塞主请求链路。
5. 健康检查、测速和余额查询应与请求代理解耦。

## 12. MVP 范围

### 12.1 MVP 必做

| 功能 | 优先级 | 说明 |
|------|--------|------|
| Provider CRUD | P0 | 支持 auth_mode 和 extra_headers |
| 本地 Gateway | P0 | 支持 `/v1/*` 透传和 stream |
| 活跃 Provider 切换 | P0 | UI 与托盘均可操作 |
| 健康检查 | P0 | 单个 Provider 基础可用性验证 |
| 请求日志 | P0 | 元数据、错误摘要、筛选 |
| 保守错误分类 | P0 | 先做通用层 |
| 基础兼容性矩阵 | P0 | 至少覆盖 2 类客户端路径 |

### 12.2 MVP 可选

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 真实测速 | P1 | 可能消耗 token |
| 托盘快速切换 | P1 | 桌面体验增强 |
| 模型列表 | P1 | 仅对支持的 Provider |
| 余额查询 | P1 | 只做 2-3 家适配 |
| Provider 导入导出 | P1 | JSON 文件 |

### 12.3 MVP 不做

1. 自动 fallback
2. 订阅协议
3. 自动配置用户工具
4. 云同步
5. 团队共享
6. 模型级路由

## 13. 里程碑

### M1：核心可用

范围：

1. Provider CRUD
2. 本地安全存储接入
3. Gateway 基础透传
4. 活跃 Provider 切换
5. 基础 UI

验收标准：

1. 用户能添加两个 Provider。
2. 用户能在 UI 中切换活跃 Provider。
3. Claude Code 能通过 `/v1/messages` 成功请求。
4. 一个 OpenAI-compatible 客户端能通过 `/v1/chat/completions` 成功请求。
5. Stream 场景可正常工作。

### M2：可排障

范围：

1. 健康检查
2. 请求日志
3. 通用错误分类
4. 托盘切换

验收标准：

1. 用户请求失败后，能在 10 秒内定位为 auth、network、timeout、rate limit 或 upstream error 中的一类。
2. 用户可以从托盘快速看到当前活跃 Provider 并完成切换。

### M3：可内测

范围：

1. 基础测速
2. 模型列表
3. 2-3 家 Provider 余额适配
4. Provider 导入导出
5. 兼容性矩阵补充

验收标准：

1. 至少形成一份真实可维护的兼容性矩阵。
2. 至少有 5-10 个真实用户完成连续使用反馈。

---

## 14. 成功标准

MVP 是否成功，主要看以下指标：

1. 新用户能在 3 分钟内完成首次接入。
2. 切换 Provider 的操作耗时低于 3 秒。
3. 用户在接入后无需再频繁修改各工具配置。
4. 请求失败时，用户能快速定位问题大类。
5. 至少 5 个真实用户在一周内持续使用。

## 15. 核心风险与应对

### 风险 1：兼容性不像文档设想得那么统一

应对：

1. 建立兼容性矩阵。
2. Provider 引入 `auth_mode` 和 `extra_headers`。
3. 将“支持”定义为“验证通过”，而不是“理论可透传”。

### 风险 2：错误分类准确率不稳定

应对：

1. 首版只承诺保守的通用分类。
2. 更细分类只对适配 Provider 开启。
3. UI 展示“推断”与“确定”的区别。

### 风险 3：测速与健康检查引起配额消耗

应对：

1. 默认先做健康检查。
2. 真实测速显式提示可能消耗 token。
3. 不做高频自动批量测速。

### 风险 4：本地存储敏感信息不安全

应对：

1. 使用系统安全存储。
2. 日志不记录敏感 body。
3. Gateway 默认仅监听 localhost。

### 风险 5：用户认为“手动切一下也够用”

应对：

1. 强调统一入口和故障排查价值。
2. 优先把日志、健康检查和切换体验做扎实。
3. 不把产品宣传成“更快改配置”，而是“更稳定地管理多 Provider”。

## 16. 后续版本方向

以下能力不进入首版范围，待 MVP 验证后再推进：

1. 自动 fallback
2. Provider 分组和标签
3. 模型级切换
4. 更多 Provider adapter
5. 配置订阅协议
6. 常见工具接入助手
7. 配置备份与同步

---

## 17. 结论

Clash for AI 首版应该被定义为一个“保守、稳定、可观测”的本地多 Provider 管理工具。

它的第一优先级不是覆盖所有协议和所有 Provider，而是把以下闭环做扎实：

1. 配置一次 localhost
2. 管理多个 Provider
3. 随时切换
4. 看到状态
5. 出错能定位大类

如果这个闭环成立，后续再扩展余额、兼容性矩阵、自动 fallback 和订阅生态才有意义。
